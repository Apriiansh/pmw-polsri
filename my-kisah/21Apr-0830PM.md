# Kisah: Personalisasi Dashboard & Integrasi Data Real-Time

## Status Saat Ini
- **Controller**: `Dashboard.php` telah direfaktorisasi sepenuhnya.
- **Data Source**: Menggunakan `PmwProposalModel`, `PmwPeriodModel`, `LecturerModel`, dan `PmwGuidanceLogbookModel`.
- **UI Consistency**: Data mapping di Controller memastikan kunci array sesuai dengan ekspektasi view `dashboard/index.php`.

## Apa yang Diimplementasikan?

### 1. Deteksi Periode Aktif
Sistem kini secara otomatis mencari periode PMW yang sedang aktif melalui `PmwPeriodModel::getActive()`. Semua statistik Admin kini difilter berdasarkan periode ini untuk akurasi data tahun berjalan.

### 2. Dashboard Role-Based
- **Admin**: Menampilkan total estimasi pendanaan (RAB), jumlah tim lolos, success rate, dan antrean proposal yang butuh review.
- **Mahasiswa**: Fokus pada status proposal tim sendiri, progress pengerjaan, dan jumlah bimbingan yang telah divalidasi dosen.
- **Dosen**: Menampilkan beban bimbingan (jumlah tim) dan jumlah logbook bimbingan yang menunggu validasi (pending).

### 3. Sinkronisasi Status & UI
Label status di database dipetakan ke label yang dipahami oleh View:
- `approved` -> **Disetujui** (Hijau)
- `submitted`/`draft` -> **Review** (Biru)
- `revision` -> **Revisi** (Kuning)
- `rejected` -> **Ditolak** (Merah)

### 4. Quick Actions
Tautan "Aksi Cepat" telah disesuaikan dengan role masing-masing:
- Admin: Manajemen User, Atur Jadwal, Ekspor Laporan.
- Mahasiswa: Kelola Proposal, Catat Bimbingan.
- Dosen: Monitoring Tim, Validasi Logbook.

## Dampak Teknis
- Kode lebih maintainable karena logika data dipisahkan per role.
- Tidak ada lagi data hardcoded; dashboard kini mencerminkan kondisi database sebenarnya.
- Performa terjaga dengan penggunaan query builder CI4 yang efisien.

## Langkah Selanjutnya
- Implementasi query spesifik untuk role **Reviewer** dan **Mentor** setelah skema penugasan (assignment) mereka difinalisasi di database.
- Optimasi caching untuk dashboard statistik jika jumlah data proposal mencapai ribuan.
