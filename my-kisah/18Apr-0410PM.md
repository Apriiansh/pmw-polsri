# Kisah: Enforcing One-to-One Assignments

## 18 April 2026 - 04:10 PM

### Masalah
Ditemukan inkonsistensi data di mana satu Dosen Pembimbing (Lecturer) dapat terhubung ke lebih dari satu tim (proposal). User menginginkan aturan ketat: **1 Dosen/Mentor = 1 Tim**.

### Perubahan yang Dilakukan
1.  **Migrasi Database**: Membuat migrasi `2026_04_18_160700_EnforceUniqueProposalAssignments` untuk menambahkan `UNIQUE INDEX` pada kolom `lecturer_id` dan `mentor_id` di tabel `pmw_proposal_assignments`.
2.  **Pembersihan Data**: Mengidentifikasi duplikasi data (Dosen ID 1 terhubung ke Proposal ID 1 dan 4). Melakukan update manual pada Proposal ID 4 untuk dipindahkan ke Dosen ID 4 yang masih tersedia.
3.  **Model Improvements**: Menambahkan method `getAvailable()` pada `LecturerModel` dan `MentorModel` untuk memudahkan filter pencarian dosen/mentor yang belum memiliki tim di masa mendatang.
4.  **Verifikasi**: Menjalankan query audit untuk memastikan tidak ada lagi ID dosen atau mentor yang muncul lebih dari satu kali di tabel assignment.

### Hasil
Sekarang sistem secara otomatis akan menolak (Database Level Error) jika ada upaya untuk memasangkan dosen atau mentor yang sudah memiliki tim ke tim lain. Data yang ada telah dibersihkan dan mematuhi aturan baru.

### Catatan Keamanan
*   Constraint unik ini memungkinkan nilai `NULL` (Dosen/Mentor yang belum ditugaskan).
*   Jika di masa depan diperlukan penugasan multi-periode, tabel assignment perlu menyertakan `period_id` dalam index uniknya. Untuk saat ini, aturan berlaku global sesuai permintaan user.
